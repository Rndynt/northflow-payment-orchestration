# Critical Design Constraint

Existing API credentials are stored as prefix + one-way hash. A one-way hash cannot be used by the server to recompute an HMAC signature.

Therefore S9.4 must not pretend that the existing hashed credential secret can verify signed requests.

Implement signed requests using a new explicit signing-key model.

Required distinction:

```txt
ClientCredential = existing bearer credential used for API key auth
ClientSigningKey = new HMAC signing key used for request signatures
```

The signing secret must be generated by Northflow, returned only once, and stored encrypted at rest or otherwise protected in a way that lets the server verify HMAC signatures.

If encryption-at-rest is implemented, use a dedicated env master key. Do not log it. Do not return it in health/version/ready.

# Part H - Signed Auth Middleware

Extend auth middleware to support signed requests.

Resolution flow in `optional` mode:

```txt
1. If signed headers are present, attempt signed request auth.
2. If signed auth succeeds, attach req.auth and continue.
3. If signed headers are present but invalid, reject. Do not silently fall back to bearer auth.
4. If no signed headers are present, fall back to existing bearer/API key auth.
```

Resolution flow in `required` mode:

```txt
1. Protected /v1 routes require valid signed request auth.
2. Missing signed headers -> 401 SIGNED_REQUEST_REQUIRED.
3. Bearer-only auth is rejected for protected /v1 routes except explicit exemptions.
```

Rules:

- Signed auth must populate the same `req.auth` shape used by bearer auth.
- Merchant access and scope guards must remain unchanged after auth.
- Do not bypass rate limiting.
- Rate limiting should still identify authenticated clientId after signed auth.
- `lastUsedAt` must update on signing key after successful signed auth.
- Nonce must be consumed atomically after signature and timestamp are valid.
- If nonce already used, reject replay.
- If nonce consume fails due to storage error, fail closed for signed auth.
